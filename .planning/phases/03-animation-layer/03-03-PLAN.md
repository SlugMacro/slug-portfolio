Modify `src/components/home/ProjectCard.tsx`:

1. Add import: `import { motion } from "motion/react"`.

2. Replace the outer `<button>` element with `<motion.button>`. Keep all existing props and className exactly the same. Add:
   - `whileHover={{ scale: 1.015, boxShadow: "0 8px 30px rgba(0,0,0,0.08)" }}`
   - `whileTap={{ scale: 0.99 }}`
   - `transition={{ duration: 0.2, ease: "easeOut" }}`

   The result:
   ```tsx
   <motion.button
     onClick={onClick}
     className="group block w-full cursor-pointer border-b border-border bg-bg pb-12 pt-6 text-left md:px-6"
     whileHover={{ scale: 1.015, boxShadow: "0 8px 30px rgba(0,0,0,0.08)" }}
     whileTap={{ scale: 0.99 }}
     transition={{ duration: 0.2, ease: "easeOut" }}
   >
   ```

3. Keep the closing `</motion.button>` instead of `</button>`.

4. Do NOT change any inner content. The arrow `<span>` with `group-hover:opacity-100` CSS transition stays as-is -- it works alongside Framer Motion's `whileHover` because CSS `:hover` and Framer's hover detection both fire on the same user action.

Why scale: 1.015 (not larger): The cards are big -- even 1.5% scale is noticeable. Larger values (1.03+) feel exaggerated on big surfaces. The box-shadow adds depth without being heavy (0.08 alpha = very subtle).

Why whileTap: 0.99: A tiny press-down on click gives tactile "button pressed" feedback. Combined with the scale-up on hover, it creates a satisfying click sequence: hover(up) -> click(down) -> release(up).

Important: Only animate `transform` (scale) and `boxShadow` via whileHover. Never animate `width`, `height`, or `padding` per the architecture decision. boxShadow is safe to animate because Framer Motion composites it efficiently.

Why motion.span not motion.div: The label is inline text content. A `<span>` is semantically correct. Using `motion.div` would introduce a block element inside an inline context.
